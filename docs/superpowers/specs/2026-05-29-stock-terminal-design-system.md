# 股票终端设计系统重构 - 设计文档

## 概述

为现有 Chrome 扩展「赚钱助手」建立一套统一的交易终端设计系统，目标不是“更花”，而是“更快看盘”。

这次重构的核心不是单独美化某一个页面，而是把 `popup`、`options`、`floating overlay`、股票/基金详情、评估中心、通知面板统一到同一套视觉语言、信息层级和交互规则中，让用户在 3 秒内完成一次有效扫盘：

1. 先看市场环境
2. 再看持仓变化
3. 再看异常和告警
4. 最后进入详情确认

## 设计目标

### 主目标

- 提高扫盘速度
- 降低认知切换成本
- 让核心数字和状态一眼可见
- 让列表、详情、设置使用同一套设计语法

### 次目标

- 保留 A 股习惯的涨跌语义
- 兼容浅色、深色、白色三种主题
- 支持高密度信息展示但不失层级
- 为后续新模块提供可复用 token 和组件边界

### 不做的事

- 不引入重装饰风格
- 不把页面做成“富媒体仪表盘”
- 不把每个模块设计成独立视觉系统
- 不为视觉而牺牲列表密度和操作效率

## 设计原则

### 1. 先结果，后过程

默认界面只展示决策信息，细节通过展开、弹层或详情页补充。

### 2. 以表格为骨架

这是一套高频扫盘工具，表格不是附属，而是主结构。图表、卡片、弹层都围绕表格服务。

### 3. 数字优先

所有金额、涨跌幅、仓位、收益、比例都使用统一的数字规范、对齐方式和语义色。

### 4. 少而强的颜色

颜色只负责强调状态，不承担大面积装饰。界面背景保持克制，红绿只在结果层出现。

### 5. 同一套组件语义贯穿全局

按钮、标签、表格、弹层、输入框、开关、卡片，在 popup 和 options 中使用同样的视觉规则。

## 系统结构

### 页面职责

| 页面 | 职责 | 设计重点 |
|---|---|---|
| `popup` | 高频扫盘主界面 | 密度、排序、状态、快速判断 |
| `options` | 系统控制台 | 配置、告警、导入导出、隐私 |
| `floating overlay` | 轻量随手看盘 | 小体积、单任务、低干扰 |
| 详情弹层 | 深入确认 | 图表、指标、决策摘要 |
| 通知面板 | 异常和提醒 | 严重度、时间、可追踪性 |
| 评估/风控 | 辅助决策 | 评分、风险、建议动作 |

### 信息层级

1. 市场总览
2. 自选与持仓列表
3. 异常、告警、信号
4. 个股/基金/指数详情
5. 配置和系统控制

## Token 体系

采用三层 token：

1. Primitive：原始值
2. Semantic：语义别名
3. Component：组件级别映射

## 主题策略

### 深色

- 作为默认工作主题
- 适合高频盯盘和交易时段
- 背景采用冷色渐变，但不抢内容

### 浅色

- 适合白天和轻阅读场景
- 保留相同层级逻辑
- 保持红绿语义不变

### 白色

- 适合打印感、轻办公感
- 更强调边框、分隔和清晰度
- 降低背景干扰，突出列表结构

## 布局系统

### 1. Popup 主界面

#### 结构

```
┌─────────────────────────────────────────────────────┐
│ 市场摘要条                                           │
├──────┬──────────────────────────────────────────────┤
│ 侧边 │ 主内容区                                      │
│ 导航 │  - 列表/面板                                  │
│      │  - 详情/弹层                                  │
└──────┴──────────────────────────────────────────────┘
```

#### 规则

- 左侧导航固定窄宽
- 中间主区只保留一条阅读主线
- 顶部摘要条固定在信息首位
- 详情弹层不改变主阅读方向

### 2. Options 页面

#### 结构

- 顶部标题与状态
- 中部按分组排列配置
- 底部放导入导出、备份、重置类高级操作

#### 规则

- 一个分区只负责一类配置
- 每个配置项都需要一句说明
- 风险操作必须有确认

### 3. Floating Overlay

#### 规则

- 只展示少量关键指标
- 适合一眼判断，不适合编辑
- 默认不抢桌面焦点

### 4. 详情弹层

#### 规则

- 顶部放标识和核心价格
- 中部放图表和诊断
- 底部放动作和辅助信息
- 弹层宽度受控，不做全屏大页

## 核心组件规范

### 1. SideNav

#### 职责

- 模块切换
- 主题切换
- 设置入口
- 市场状态小卡片

#### 规范

- 图标在上，文字在下
- 激活态必须显著
- 未读数徽标必须靠近对应入口
- 底部统计只能做辅助，不可压过主导航

### 2. Market Stats Strip

#### 职责

- 输出市场方向、成交、家数和量能状态

#### 规范

- 单项结构统一：标签 + 数值
- 涨跌用 `up/down` 语义
- 弱化项使用更小字号

### 3. Data Table

#### 职责

- 承载股票和基金的主扫盘逻辑

#### 规范

- 表头固定语义
- 排序列必须有视觉反馈
- 行高统一
- 允许拖拽、置顶、快捷编辑
- 选中、拖拽、编辑状态不能互相冲突

#### 行结构

每行应遵循：

1. 身份信息
2. 核心图形
3. 关键数值
4. 次级数值
5. 操作区

### 4. Dual Value Cell

#### 职责

- 展示上下两层数值，例如金额 + 比例、成本 + 现价

#### 规范

- 上层是主值，下层是解释值
- 主值字号更大，解释值更小
- 对齐方式必须一致，避免视觉漂移

### 5. Tag / Badge

#### 职责

- 标记身份、状态、信号和风险

#### 规范

- 标签数量上限应受控
- 只保留最影响判断的 1 到 2 个标签
- 额外标签用折叠数展示
- 正/负/警示标签必须语义明确

### 6. Buttons

#### 职责

- 触发切换、刷新、保存、关闭等动作

#### 规范

- 主按钮只用于当前最关键动作
- 次按钮用于辅助动作
- 危险按钮必须有明确风险语义
- 图标按钮需要可访问名称

### 7. Modal / Drawer

#### 职责

- 承接详情、确认和辅助分析

#### 规范

- 遮罩要轻，不压死背景
- 标题区必须固定
- 关闭按钮位置统一
- 不同详情类型复用相同壳体

### 8. Chart Shell

#### 职责

- 包裹 K 线、分时图、雷达图、资产曲线等视觉图表

#### 规范

- 图表容器必须有统一边框和留白
- 图表颜色来自语义 token
- 不允许图表直接使用硬编码主题色

## 组件状态规范

### 导航状态

| 状态 | 表现 |
|---|---|
| 默认 | 中性文字和轻底色 |
| Hover | 边界更明显，底色略抬高 |
| Active | 强底色、强文字、轻阴影 |
| Disabled | 降低对比度，不可点击 |

### 表格状态

| 状态 | 表现 |
|---|---|
| 默认 | 清晰分隔 |
| Hover | 整行高亮 |
| Selected | 明显激活底色 |
| Dragging | 轻微透明和位移感 |
| Editing | 输入框替代单元格 |
| Pinned | 置顶标识和固定语义 |

### 数值状态

| 状态 | 表现 |
|---|---|
| 正收益 | `up` 语义 |
| 负收益 | `down` 语义 |
| 零值 | 中性文本 |
| 缺失 | `--` 或占位符 |
| 隐私隐藏 | `***` |

## 交互规范

### 1. 扫盘路径

默认阅读顺序：

1. 顶部市场摘要
2. 侧边模块切换
3. 当前列表排序和异常标记
4. 行内关键数值
5. 详情弹层中的图表和诊断

### 2. 排序

- 排序必须可见
- 当前排序字段高亮
- 升序和降序图标要易识别
- 一键恢复默认排序应可用

### 3. 编辑

- 单元格点击可直接编辑
- `Enter` 提交
- `Esc` 取消
- 失焦后自动校验
- 编辑时只替换当前格，不破坏整行布局

### 4. 刷新

- 刷新中要有显性反馈
- 手动刷新和后台刷新都应可见
- 数据更新不要导致全页跳动

### 5. 详情

- 点击行进入详情
- 详情优先承接“确认判断”
- 返回后保持列表滚动位置和排序状态

## 动效规范

动效只做“辅助定位”，不做表演。

### 建议使用的动效

- 页面切换淡入淡出
- 行 hover 的轻抬升
- 弹层打开的缩放进入
- 数据更新的短暂强调

### 禁止事项

- 夸张弹跳
- 大面积闪烁
- 长时间旋转
- 影响阅读路径的连续动画

## 可访问性

- 所有交互图标必须有可理解的标题或 aria label
- 键盘必须能完成主要路径
- 焦点态必须可见
- 颜色不能作为唯一信息载体
- 数字和状态要同时提供文本表达

## 图表规范

### 统一要求

- 使用同一套语义色
- 图表外壳与列表外壳一致
- 图表对比色不超过 3 种主色
- 重要数值必须配文字说明

### 建议图表类型

| 场景 | 图表 |
|---|---|
| 分时走势 | 线图 |
| K 线详情 | 蜡烛图 |
| 仓位结构 | 圆环/条形 |
| 风险评估 | 雷达图 |
| 资产变化 | 折线图 |

## 迁移策略

建议按三阶段落地：

### Phase 1：统一基础 token

- 抽出颜色、字体、间距、圆角、阴影 token
- 让 popup、options、overlay 使用统一语义
- 先不改业务逻辑

### Phase 2：统一高频组件

- SideNav
- 表格
- 标签
- 按钮
- 弹层
- 输入框

### Phase 3：重排页面信息层级

- 重做 popup 信息顺序
- 统一详情弹层
- 优化 options 的分组和控制台风格
- 收束浮窗和通知的视觉语言

## 验收标准

这套设计算成功，至少要满足以下条件：

1. 用户打开 popup 后能明显感受到“这是一个交易终端，而不是普通列表页”
2. 市场状态、持仓状态、异常状态的层级一眼能分清
3. 股票和基金表格使用相同的数值和状态语法
4. 深色、浅色、白色主题在结构上保持一致
5. 详情弹层不会破坏主界面的扫盘节奏
6. 设置页看起来像控制台，而不是杂乱的配置堆
7. 颜色、间距、圆角和阴影不再散落在各个文件里各自定义

## 风险与约束

- 当前代码里已经有大量局部样式，迁移时要避免一次性打散所有布局
- popup 高度和宽度受扩展窗口限制，设计不能依赖大屏宽裕空间
- 有些视觉优化会影响信息密度，必须优先保留扫盘速度
- 主题切换逻辑必须和现有 `displayConfig`、`theme` 兼容

## 结论

这次重构的本质，是把现有“功能很多的扩展”提升为“有统一设计语法的交易终端”。

如果后续按这份规范执行，优先级应该是：

1. token
2. 表格和导航
3. 详情和弹层
4. 设置页和浮窗

这样可以最小成本地把“更快看盘”落实到每一次打开扩展时的实际体验里。
