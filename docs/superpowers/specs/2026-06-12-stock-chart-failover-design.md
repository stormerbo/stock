# 股票分时与 K 线多源自动切换设计

## 背景

当前项目的股票图表能力主要依赖腾讯链路：

- 股票详情页分时图
- 股票详情页 K 线
- 技术分析中的日 K 数据
- 部分缓存与衍生计算

近期腾讯侧图表接口出现明显限流或不稳定现象，导致股票详情页和依赖 K 线的功能可用性下降。项目内虽然已经存在部分东方财富图表接口使用经验，但尚未形成统一的“主备切换”机制。

本次目标不是简单把某个接口硬切到另一家，而是为股票分时与 K 线建立一套多源自动切换机制，在单个数据源限流、超时、空响应或结构异常时自动降级到备用源。

## 目标

- 股票分时图支持多源自动切换。
- 股票 K 线支持多源自动切换。
- 默认优先级为 `东财优先，腾讯兜底`。
- 详情页、技术分析、缓存层尽量走统一的图表数据入口。
- 主源失败时优先静默切备用，不直接向用户暴露错误。

## 非目标

- 不重做现有图表 UI。
- 不引入 WebSocket 方案。
- 不一次性重构所有行情抓取代码。
- 不在首版加入复杂监控面板或用户可配置的数据源开关。

## 方案概览

推荐方案是在共享层增加“股票图表多源调度器”，把分时和 K 线请求统一包装成：

- 先请求主源（东财）
- 若失败或命中异常特征，则自动请求备用源（腾讯）
- 仅当主备都失败时，才把错误返回给上层页面

与“直接全量切东财”相比，这个方案更稳：

- 东财可作为主链路提升当前可用性
- 腾讯仍保留为兼容 fallback
- 后续若还要引入第三源，也可以在调度层扩展

## 数据源优先级

首版固定优先级：

1. 东方财富
2. 腾讯

原因：

- 当前痛点主要来自腾讯链路限流
- 项目中已经有多处东财接口使用经验
- 东财 HTTP 接口更适合当前 Chrome 扩展结构，无需额外网关

## 适用范围

本次多源切换覆盖：

- 股票详情页分时图
- 股票详情页 K 线
- `technical-analysis` 中日 K 获取
- 后续可以顺带接入 `kline-cache` 相关链路

不要求一次性替换所有股票快照行情抓取，重点先覆盖图表链路。

## 异常判定规则

触发主源降级到备用源的条件包括：

- 网络请求失败
- 请求超时
- 返回空数组
- 返回关键字段缺失
- JSON 结构与预期不符
- 命中明显限流/风控特征

“明显限流/风控特征”首版可以按保守规则实现：

- HTTP 成功但数据为空且非停牌/非休市合理场景
- 返回固定错误文案或异常包结构
- 同一接口短时间连续失败

## 熔断与退避

为避免在限流状态下反复打同一主源，建议加轻量熔断：

- 为每个图表源维护短时失败状态
- 某源连续失败达到阈值后，在短窗口内暂时降级优先级
- 例如连续失败 3 次后，5 分钟内先走备用源

这不是严格的分布式熔断实现，只需要做到“减少无效重试、避免用户连续碰到坏源”。

## 共享层设计

建议新增统一调度函数，而不是让每个调用方自己写 try/catch：

- `fetchStockIntradayWithFallback(code, options?)`
- `fetchStockKlineWithFallback(code, period, options?)`
- `fetchDayFqKline(...)` 内部也尽量复用统一日 K 入口

每个统一入口都返回：

- 规范化后的图表数据
- 实际命中的数据源标识（可选）

这样上层如果以后要做调试、埋点、日志，就有地方挂。

## 数据结构统一

不同数据源返回结构不同，调度层上方不应该感知差异。

需要统一的输出包括：

### 分时图

- 时间点数组
- 价格数组
- 前收/基准价

### K 线

- 日期
- 开盘
- 收盘
- 最高
- 最低
- 成交量

上层图表组件继续只吃统一结构，不关心数据来自东财还是腾讯。

## 调用方改造

### 股票详情页

`stockDetail.ts` 中目前直接请求腾讯分时与腾讯/东财混合 K 线的逻辑，改为优先通过共享调度器取数据。

### 技术分析

`technical-analysis.ts` 中的 `fetchDayFqKline(...)` 不再直接绑定腾讯，而是改为“日 K 多源入口”。

### 缓存层

若 `kline-cache` 当前仍绕开共享入口，建议一起切到统一 K 线入口，避免不同模块出现不同可用性表现。

## 错误处理

页面层策略：

- 主源失败但备用成功：页面无感知
- 主备都失败：显示现有错误态
- 如果备用成功但数据量较少，仍优先展示可用数据，不阻塞页面

日志层策略：

- 开发日志可打印主源失败、备用接管信息
- 不需要把“主源失败但备用成功”直接暴露给普通用户

## 测试策略

至少需要补以下测试：

### 共享层测试

- 主源成功时不访问备用源
- 主源失败时切到备用源
- 主源空数据时切到备用源
- 主源命中熔断窗口时直接跳备用源
- 主备都失败时返回错误

### 解析测试

- 东财分时解析输出正确
- 东财 K 线解析输出正确
- 腾讯 fallback 解析保持兼容

### 集成验证

- 股票详情页分时正常展示
- 股票详情页日/周/月 K 线正常展示
- 技术分析功能仍可用
- `npm run build` 通过

## 文件边界建议

建议新增或扩展以下文件：

- `src/shared/stock-chart-sources.ts`
  - 各数据源的分时/K线抓取器
- `src/shared/stock-chart-failover.ts`
  - 主备切换、熔断、重试、异常判定
- `src/popup/stockDetail.ts`
  - 改走统一图表入口
- `src/shared/technical-analysis.ts`
  - 日 K 入口接入 failover
- `src/shared/kline-cache.ts`
  - 如有必要，统一接入 K 线入口

若实现时发现拆文件成本过高，也可先以内聚方式落在共享层，但仍要保证“源适配”和“切换策略”逻辑分开。

## 风险与注意点

- 东财分时与 K 线接口在字段和时间维度上与腾讯不同，解析层必须统一规范。
- 如果 failover 分散在多个文件里临时拼接，后续维护会非常痛苦，必须尽量集中到共享入口。
- 熔断窗口不能过长，否则单次误判会让主源长期失效；但也不能没有，否则限流时会反复触发。
- 部分功能如果仍直接调用腾讯旧接口，会出现“详情页好了，技术分析还坏着”的割裂体验。

## 结论

本次采用“东财主源 + 腾讯备用 + 共享调度器 + 轻量熔断”的方案，为股票分时与 K 线建立自动切换能力。这样能在不重做 UI 的前提下，提高图表可用性，并为后续继续接入更多数据源保留清晰扩展点。
